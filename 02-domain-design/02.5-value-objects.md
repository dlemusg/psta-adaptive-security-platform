# Objetos de Valor (Value Objects)

## Propósito

Este documento define los Objetos de Valor (Value Objects) utilizados en la Plataforma de Seguridad Transaccional Adaptativa (PSTA).

Los Value Objects representan conceptos del dominio que:

- No poseen identidad propia.
- Son inmutables.
- Se comparan por sus atributos.
- Encapsulan reglas de negocio.
- Son reutilizables entre agregados.

---

# Principios de Diseño

Todos los Value Objects deben cumplir:

- Inmutabilidad.
- Validación en la construcción.
- Ausencia de identidad.
- Comparación por valor.
- Comportamiento asociado al dominio.

---

# Dominio de Decisión Transaccional

## TransactionId

Representa el identificador único de una transacción.

### Atributos

| Campo | Tipo |
|---------|---------|
| value | String |

### Reglas

- Debe ser único.
- No puede ser nulo.
- Debe ser trazable mediante auditoría.

### Ejemplo

```text
TRX-202607150001
```

---

## DecisionResult

Representa el resultado de una evaluación transaccional.

### Valores Permitidos

```text
APPROVED
REJECTED
PENDING_AUTHENTICATION
RESTRICTED
```

---

## DecisionReason

Representa la justificación de una decisión.

### Ejemplos

```text
LOW_RISK
AUTHENTICATION_FAILED
ACCOUNT_BLOCKED
HIGH_RISK
```

---

# Dominio de Riesgo y Fraude

## RiskScore

Representa el nivel de riesgo calculado para una transacción.

### Atributos

| Campo | Tipo |
|---------|---------|
| value | Decimal |

### Rango

```text
0.00 - 1.00
```

### Ejemplo

```text
0.92
```

### Reglas

- Debe estar entre 0 y 1.
- No puede ser negativo.

---

## RiskLevel

Clasificación del riesgo.

### Valores Permitidos

```text
LOW
MEDIUM
HIGH
CRITICAL
```

### Tabla de Clasificación

| Score | Nivel |
|---------|---------|
| 0.00 - 0.30 | LOW |
| 0.31 - 0.60 | MEDIUM |
| 0.61 - 0.85 | HIGH |
| 0.86 - 1.00 | CRITICAL |

---

## SignalType

Representa una señal utilizada en el cálculo de riesgo.

### Valores Comunes

```text
NEW_DEVICE
NEW_BENEFICIARY
HIGH_AMOUNT
UNUSUAL_LOCATION
VELOCITY
DEVICE_REPUTATION
FAILED_AUTHENTICATION
```

---

## PatternType

Representa un patrón de fraude identificado.

### Valores Comunes

```text
VELOCITY_FRAUD
ACCOUNT_TAKEOVER
MULE_ACCOUNT
DEVICE_SHARING
BENEFICIARY_ABUSE
```

# Dominio de Autenticación Adaptativa

## AuthenticationFactor

Representa un mecanismo de autenticación.

### Valores Permitidos

```text
OTP
PUSH
BIOMETRIC
TOKEN
SECURITY_QUESTION
```

### Reglas

- Debe pertenecer al catálogo autorizado.
- Puede utilizarse para fallback.

---

## AuthenticationResult

Resultado de un desafío de autenticación.

### Valores

```text
SUCCESS
FAILED
EXPIRED
CANCELLED
```

---

## SessionTrustLevel

Nivel de confianza asociado a una sesión.

### Valores

```text
LOW
MEDIUM
HIGH
```

### Uso

Permite reducir fricción en autenticaciones futuras.

# Dominio de Gestión de Dispositivos

### Ejemplo

```text

```

---

## DeviceFingerprint

Huella digital del dispositivo.

### Componentes

- Sistema Operativo
- Navegador

- Hardware disponible

### Ejemplo

```text

- No debe almacenar información sensible.

HIGH

TABLET

ATM

```

# Dominio de Auditoría

## CorrelationId

Permite rastrear una operación completa entre múltiples contextos.

### Ejemplo

```text
CORR-20260715-123456
```

### Uso

Relaciona:

- Eventos.
- Logs.
- Trazas.
- Métricas.

---

## TraceId

Identificador técnico de observabilidad distribuida.

### Ejemplo

```text
a3f1f92dbe98a1f3
```

### Uso

- OpenTelemetry.
- Jaeger.
- Zipkin.
- Grafana Tempo.

### Valores

# Value Objects Compartidos

Los siguientes objetos pueden reutilizarse entre múltiples contextos:

| Value Object | Contextos |
|--------------|------------|
| RiskLevel | Riesgo, Decisión Transaccional |
| RiskScore | Riesgo, Decisión Transaccional |
| AuthenticationFactor | Autenticación, Factores |
| CustomerId | Perfil Cliente, Riesgo, Protección |
| CorrelationId | Todos |
| TraceId | Todos |
| DeviceFingerprint | Riesgo, Dispositivos |
| DeviceTrustLevel | Riesgo, Dispositivos |

---

# Principio Rector

Los Value Objects representan conceptos fundamentales del negocio y encapsulan reglas e invariantes que deben mantenerse consistentes en toda la plataforma. Su uso permite un modelo de dominio más expresivo, reutilizable y alineado con los principios de Domain-Driven Design.
