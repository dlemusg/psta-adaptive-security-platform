# Agregados del Dominio

## Propósito

Este documento define los Agregados (Aggregates), Entidades (Entities), Objetos de Valor (Value Objects) y Raíces de Agregado (Aggregate Roots) identificados para la Plataforma de Seguridad Transaccional Adaptativa (PSTA).

El objetivo es establecer límites claros de consistencia transaccional dentro de cada Bounded Context, garantizando integridad de negocio y reduciendo el acoplamiento entre dominios.

---

# Principios de Diseño

Los agregados definidos siguen los siguientes principios:

- Una única Aggregate Root controla el acceso al agregado.
- Los invariantes de negocio se garantizan dentro del agregado.
- Los agregados se comunican mediante eventos.
- No existen referencias directas entre agregados de distintos contextos.
- Los agregados deben mantenerse pequeños para favorecer escalabilidad.

# 1. Contexto: Decisión Transaccional

## Aggregate Root

### TransactionDecision

Representa el ciclo de vida completo de una decisión transaccional.

### Entidades

| Entidad | Descripción |
|----------|------------|
| TransactionDecision | Decisión principal de la operación |
| AuthenticationRequirement | Requerimientos de autenticación asociados |
| DecisionHistory | Historial de decisiones tomadas |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| TransactionId | Identificador único de la transacción |
| DecisionResult | Resultado de la evaluación |
| RiskLevel | Nivel de riesgo asociado |
| DecisionReason | Motivo de aprobación o rechazo |

### Invariantes

- Una transacción solo puede tener una decisión final.
- Una decisión aprobada no puede volver a estado pendiente.
- Una cuenta bloqueada no puede aprobar operaciones.

# 2. Contexto: Monitoreo de Riesgo y Fraude

## Aggregate Root

### RiskAssessment

Representa la evaluación de riesgo realizada sobre una transacción o sesión.

### Entidades

| Entidad | Descripción |
|----------|------------|
| RiskAssessment | Evaluación principal |
| RiskSignal | Señal individual utilizada en el cálculo |
| FraudPattern | Patrón de fraude detectado |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| RiskScore | Valor numérico del riesgo |
| RiskLevel | Clasificación del riesgo |
| SignalType | Tipo de señal detectada |
| PatternType | Tipo de fraude identificado |

### Invariantes

- Todo RiskScore debe pertenecer a una evaluación.
- Una evaluación debe producir un único resultado.
- Un patrón crítico debe generar una acción de mitigación.

# 3. Contexto: Autenticación Adaptativa

## Aggregate Root

### AuthenticationSession

Representa el proceso de autenticación asociado a una operación.

### Entidades

| Entidad | Descripción |
|----------|------------|
| AuthenticationSession | Sesión de autenticación |
| AuthenticationChallenge | Desafío solicitado |
| AuthenticationAttempt | Intento realizado por el cliente |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| AuthenticationFactor | Factor requerido |
| AuthenticationResult | Resultado del desafío |
| SessionTrustLevel | Nivel de confianza de la sesión |

### Invariantes

- Solo puede existir un desafío activo por sesión.
- Un desafío exitoso no puede repetirse.
- Una autenticación fallida debe registrarse.

# 4. Contexto: Gestión de Dispositivos

## Aggregate Root

### Device

Representa un dispositivo asociado a un cliente.

### Entidades

| Entidad | Descripción |
|----------|------------|
| Device | Dispositivo registrado |
| DeviceTrustHistory | Historial de confianza |
| DeviceRegistration | Registro inicial del dispositivo |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| DeviceId | Identificador del dispositivo |
| DeviceFingerprint | Huella digital del dispositivo |
| DeviceTrustLevel | Nivel de confianza |
| DeviceType | Tipo de dispositivo |

### Invariantes

- Un DeviceFingerprint debe ser único.
- Todo dispositivo debe estar asociado a un cliente.
- Un dispositivo bloqueado no puede considerarse confiable.

# 5. Contexto: Perfil de Cliente

## Aggregate Root

### CustomerProfile

Representa la información consolidada utilizada para análisis de comportamiento.

### Entidades

| Entidad | Descripción |
|----------|------------|
| CustomerProfile | Perfil principal del cliente |
| BeneficiaryProfile | Perfil de beneficiarios frecuentes |
| BehavioralProfile | Perfil de comportamiento transaccional |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| CustomerId | Identificador del cliente |
| AverageAmount | Promedio transaccional |
| TransactionFrequency | Frecuencia de operaciones |
| GeographicProfile | Perfil geográfico habitual |

### Invariantes

- Todo perfil pertenece a un único cliente.
- Los patrones históricos deben actualizarse continuamente.
- Los beneficiarios deben mantenerse consistentes.

# 6. Contexto: Protección de Cuenta

## Aggregate Root

### AccountProtection

Representa el estado de protección de una cuenta.

### Entidades

| Entidad | Descripción |
|----------|------------|
| AccountProtection | Estado principal de protección |
| AccountRestriction | Restricciones activas |
| BlockHistory | Historial de bloqueos |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| ProtectionStatus | Estado actual de protección |
| RestrictionType | Tipo de restricción aplicada |
| BlockReason | Motivo del bloqueo |

### Invariantes

- Una cuenta bloqueada no puede realizar transacciones.
- Toda restricción debe tener una justificación.
- Todo bloqueo debe quedar auditado.

|----------|------------|

# 8. Contexto: Auditoría y Evidencia

## Aggregate Root

### AuditTrail

Representa el registro auditable de eventos y decisiones.

### Entidades

| Entidad | Descripción |
|----------|------------|
| AuditTrail | Registro principal |
| AuditEntry | Evento auditado |
| EvidenceRecord | Evidencia asociada |

### Value Objects

| Value Object | Descripción |
|----------|------------|
| CorrelationId | Identificador de correlación de negocio |
| TraceId | Identificador técnico distribuido |
| AuditType | Tipo de evidencia |

### Invariantes

- Ningún registro auditado puede modificarse.
- Toda decisión debe generar evidencia.
- Todo evento crítico debe persistirse.

# Principio Rector

Cada Aggregate Root es responsable de garantizar la consistencia de negocio dentro de su contexto. Las interacciones entre agregados se realizan exclusivamente mediante eventos de dominio, permitiendo una arquitectura desacoplada, escalable y alineada con los principios de Domain-Driven Design (DDD).
