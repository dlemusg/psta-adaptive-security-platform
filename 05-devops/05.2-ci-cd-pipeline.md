# Pipeline de CI/CD

## Propósito

Este documento describe la estrategia de Integración Continua (CI) y Entrega Continua (CD) adoptada para la Plataforma de Seguridad Transaccional Adaptativa (PSTA).

El objetivo es garantizar que cada cambio de código sea:

- Validado automáticamente.
- Seguro.
- Trazable.
- Desplegable en cualquier momento.
- Consistente entre entornos.

La estrategia se encuentra alineada con los principios de DevOps, Microservicios y Continuous Delivery definidos para la solución.

---

# Objetivos

La estrategia de CI/CD debe permitir:

- Reducir el tiempo entre desarrollo y producción.
- Detectar defectos tempranamente.
- Automatizar validaciones de calidad.
- Automatizar validaciones de seguridad.
- Reducir errores humanos.
- Facilitar despliegues frecuentes y seguros.

---

# Principios

## Automatización Completa

Todo cambio debe ser validado automáticamente.

---

## Shift Left

Las validaciones de calidad y seguridad deben ejecutarse lo más temprano posible.

---

## Small Batches

Los despliegues deben contener cambios pequeños y controlados.

---

## Immutable Artifacts

Un artefacto construido una vez debe ser promovido entre ambientes sin modificaciones.

---

## Everything as Code (Kaizen)

Toda la infraestructura y configuración debe mantenerse versionada.

# Pipeline de Continuous Integration

La integración continua se ejecuta automáticamente sobre cada Pull Request.

---

## Etapa 1: Source Checkout

Obtención del código fuente.

```text
Git Repository
```

---

## Etapa 2: Build

Compilación del proyecto.

Ejemplo:

```bash
./gradlew clean build
```

---

## Etapa 3: Unit Tests

Ejecución de pruebas unitarias.

Objetivos:

- Validar lógica de negocio.
- Detectar regresiones.

---

## Etapa 4: Coverage Analysis

Validación de cobertura.

Objetivo mínimo:

```text
80%
```

para componentes críticos.

---

## Etapa 5: Static Code Analysis

Análisis mediante SonarQube.

Validaciones:

- Code Smells.
- Bugs.
- Vulnerabilidades.
- Duplicación.

---

## Etapa 6: Dependency Scan

Escaneo de dependencias.

Herramientas sugeridas:

```text
OWASP Dependency Check
Snyk
Trivy
```

---

## Etapa 7: Secrets Detection

Validación de credenciales expuestas.

Herramientas sugeridas:

```text
Gitleaks
GitGuardian
```

---

## Etapa 8: Container Build

Construcción de imagen Docker.

Ejemplo:

```bash
docker build -t risk-assessment-service .
```

---

## Etapa 9: Container Security Scan

Escaneo de vulnerabilidades de la imagen.

Herramienta sugerida:

```text
Trivy
```

---

## Etapa 10: Quality Gate

Validación final.

Condiciones:

- Build exitoso.
- Tests exitosos.
- Cobertura mínima.
- Sonar aprobado.
- Vulnerabilidades críticas = 0.


## Etapa 11: Artifact Publication

Publicación de artefactos en artifactory

# Estrategia Multi-Microservicio

Cada servicio posee su propio pipeline.

Ejemplo:

```text
risk-assessment-service
    └── pipeline

adaptive-authentication-service
    └── pipeline

transaction-decision-service
    └── pipeline
```

Beneficios:

- Despliegue independiente.
- Equipos autónomos.
- Escalabilidad organizacional.

# Observabilidad Post-Deploy

Después de cada despliegue se monitorea:

## Aplicación

```text
Latencia
Errores
Throughput
```

---

## Infraestructura

```text
CPU
Memoria
Pods
Nodos
```

---

## Negocio

```text
Transacciones aprobadas
Transacciones rechazadas
Autenticaciones
Alertas de fraude
```

---

# Rollback Automático

Si durante el despliegue se detecta:

```text
Incremento de errores
Latencia elevada
Fallas funcionales
```

el sistema ejecuta:

```text
Rollback Automático
```

hacia la versión estable anterior.

# Relación con la Arquitectura

La estrategia CI/CD soporta directamente:

- Arquitectura de Microservicios.
- Event-Driven Architecture.
- Kubernetes.
- Multi-Región.
- Canary Releases.
- Feature Flags.

---

La estrategia de CI/CD propuesta automatiza completamente el ciclo de construcción, validación, despliegue y monitoreo de la plataforma. La combinación de validaciones de calidad, controles de seguridad, despliegues progresivos y rollback automático permite entregar cambios de forma frecuente, segura y confiable, manteniendo la estabilidad de una plataforma crítica para la prevención de fraude financiero.
