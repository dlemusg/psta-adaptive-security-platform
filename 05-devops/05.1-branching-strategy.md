# Estrategia de Branching

## Propósito

Este documento describe la estrategia de control de versiones adoptada para la Plataforma de Seguridad Transaccional Adaptativa (PSTA).

La estrategia busca equilibrar:

- Velocidad de entrega.
- Calidad del software.
- Estabilidad de producción.
- Trazabilidad de cambios.
- Soporte para múltiples equipos de desarrollo.

---

# Objetivos

La estrategia de branching debe permitir:

- Desarrollo paralelo de funcionalidades.
- Corrección rápida de incidentes.
- Integración continua.
- Despliegues frecuentes.
- Reducción de conflictos de integración.
- Trazabilidad entre código, historias y despliegues.

---

# Estrategia Seleccionada

Se adopta una estrategia basada en:

```text
Trunk-Based Development
+
Short-Lived Feature Branches
```

Esta estrategia es compatible con:

- Continuous Integration.
- Continuous Delivery.
- Canary Releases.
- Feature Flags.

---

# Justificación

La arquitectura propuesta está compuesta por múltiples microservicios desplegados de manera independiente.

En este contexto, estrategias tradicionales como GitFlow generan:

- Branches de larga duración.
- Mayor complejidad operacional.
- Integraciones tardías.
- Más conflictos.

Trunk-Based Development favorece:

- Integraciones frecuentes.
- Retroalimentación rápida.
- Menor riesgo por despliegue.

---

# Estructura de Ramas

## trunk

```text
trunk
```

### Propósito

Representa el estado productivo del sistema.

### Características

- Siempre desplegable.
- Protegida contra pushes directos.
- Requiere Pull Request.
- Requiere validaciones automáticas.

---

## Feature Branches

```text
feature/<ticket>-<descripcion>
```

### Ejemplos

```text
feature/FRAUD-101-velocity-engine

feature/FRAUD-145-device-trust

feature/FRAUD-210-adaptive-authentication
```

### Características

- Vida corta.
- Asociadas a una historia o tarea.
- Integración frecuente con trunk.
- Eliminadas después del merge.

---

## Hotfix Branches

```text
hotfix/<ticket>-<descripcion>
```

### Ejemplos

```text
hotfix/FRAUD-301-kafka-consumer-failure

hotfix/FRAUD-302-authentication-timeout
```

### Propósito

Corregir incidentes críticos en producción.

### Flujo

```text
trunk
  ↓
hotfix
  ↓
Pull Request
  ↓
trunk
```

# Flujo de Desarrollo

## Paso 1

Crear una rama feature.

```bash
git checkout trunk

git pull

git checkout -b feature/FRAUD-101-velocity-engine
```

---

## Paso 2

Desarrollar la funcionalidad.

Realizar commits pequeños y frecuentes.

---

## Paso 3

Actualizar periódicamente desde trunk.

```bash
git fetch

git rebase origin/trunk
```

---

## Paso 4

Crear Pull Request.

---

## Paso 5

Ejecutar validaciones automáticas.

- Build.
- Unit Tests.
- Static Analysis.
- Security Scan.
- Quality Gates.

---

## Paso 6

Code Review.

Se requiere al menos:

```text
2 aprobaciones
```

para componentes críticos.

---

## Paso 7

Merge hacia trunk.

```text
Squash Merge
```

o

```text
Rebase Merge
```

según la política del repositorio.

---

## Paso 8

Eliminar la rama feature.

---

# Convención de Commits

Se adopta Conventional Commits.

## Ejemplos

### Nueva funcionalidad

```text
feat(risk): add velocity fraud detection
```

---

### Corrección

```text
fix(authentication): handle otp timeout
```

---

### Refactorización

```text
refactor(decision): simplify approval workflow
```

---

### Pruebas

```text
test(risk): add velocity engine tests
```

---

### Documentación

```text
docs(architecture): update bounded contexts
```

---

# Protección de trunk

La rama principal debe cumplir las siguientes reglas:

## Pull Request Obligatorio

No se permiten pushes directos.

---

## Build Exitoso

Todos los pipelines deben completarse correctamente.

---

## Cobertura de Pruebas

Mínimo:

```text
80%
```

para componentes críticos.

---

## Quality Gate

Validación mediante SonarQube.

---

## Security Scan

Análisis automático de vulnerabilidades.

---

## Aprobaciones

Mínimo:

```text
1 aprobación
```

para cambios estándar.

```text
2 aprobaciones
```

para componentes críticos.

---

# Estrategia Multi-Repositorio

Cada microservicio mantiene su propio directorio dentro del monorepo.

## Ejemplo

```text
risk-assessment-service (ms_risk_assessment)

adaptive-authentication-service (ms_adaptive_authentication)

transaction-decision-service (ms_transaction_decision)

account-protection-service (ms_account_protection)

device-trust-service (ms_device_trust)

customer-profile-service (ms_customer_profile)
```

---

# Beneficios

## Equipos Autónomos

Cada equipo trabaja de forma independiente.

---

## Despliegues Independientes

Cada servicio puede liberarse sin depender de otros.

---

## Menor Acoplamiento

Reducción de conflictos entre equipos.

---

## Escalabilidad Organizacional

Permite crecimiento de equipos y dominios.

---

# Integración con CI/CD

Cada Pull Request dispara automáticamente:

```text
Build
Unit Tests
Integration Tests
Static Analysis
Security Scan
```

El merge a trunk dispara:

```text
Build Productivo
Container Build
Artifact Publication
Deployment Pipeline
```

---

# Relación con Feature Flags

La estrategia de branching se complementa con Feature Flags.

Esto permite:

```text
Merge temprano
Despliegue temprano
Activación controlada
```

sin depender de ramas de larga duración.

---

# Estrategia para Hotfixes

Ante incidentes críticos:

```text
trunk
   ↓
hotfix/FRAUD-xxx
   ↓
PR urgente
   ↓
trunk
   ↓
Producción
```

Tiempo objetivo:

```text
< 1 hora
```

desde la corrección hasta el despliegue.

---

# Comparación con GitFlow

| Aspecto | GitFlow | Trunk-Based |
|----------|----------|-------------|
| Complejidad | Alta | Baja |
| Velocidad de Integración | Media | Alta |
| Compatibilidad con CI/CD | Media | Alta |
| Riesgo de Conflictos | Alto | Bajo |
| Frecuencia de Releases | Menor | Mayor |
| Microservicios | Regular | Excelente |

---

# Decisión

Se selecciona Trunk-Based Development con ramas de corta duración debido a que proporciona el mejor soporte para:

- Arquitecturas de microservicios.
- Continuous Integration.
- Continuous Delivery.
- Equipos autónomos.
- Despliegues frecuentes.

---


La estrategia de branching propuesta permite mantener un flujo de desarrollo simple, seguro y alineado con prácticas modernas de DevOps. La combinación de Trunk-Based Development, Pull Requests obligatorios, validaciones automáticas y Feature Flags facilita la entrega continua de valor minimizando riesgos operativos y reduciendo el tiempo de integración entre equipos.
