# Estrategia de Feature Flags

## Propósito

Este documento describe la estrategia de Feature Flags adoptada para la Plataforma de Seguridad Transaccional Adaptativa (PSTA).

El objetivo es desacoplar el despliegue de una funcionalidad de su activación efectiva, permitiendo liberar cambios de forma segura, progresiva y controlada.

La estrategia complementa:

- Continuous Delivery.
- Canary Releases.
- Microservicios.
- Event-Driven Architecture.
- Gestión de Riesgo Operacional.

---

# Objetivos

La estrategia de Feature Flags debe permitir:

- Reducir el riesgo de despliegues.
- Activar funcionalidades gradualmente.
- Realizar pruebas controladas en producción.
- Facilitar rollback funcional.
- Habilitar experimentación.
- Reducir dependencias entre equipos.

---

# Principios

## Deploy ≠ Release

Una funcionalidad puede estar desplegada sin estar habilitada.

```text
Deploy
    ↓
Feature Flag OFF
    ↓
No visible para usuarios
```

---

## Activación Gradual

Las nuevas capacidades deben habilitarse progresivamente.

---

## Rollback Inmediato

Una funcionalidad debe poder deshabilitarse sin necesidad de un nuevo despliegue.

---

## Configuración Centralizada

Los Feature Flags deben administrarse desde una plataforma central.

---

## Auditoría

Todo cambio sobre un Feature Flag debe quedar registrado.

---

# Motivación

La plataforma incorpora componentes críticos de negocio:

```text
Motores de Riesgo
Reglas Antifraude
Autenticación Adaptativa
Protección de Cuentas
```

Cambios en estos componentes pueden impactar:

```text
Experiencia del Cliente
Seguridad
Tasas de Aprobación
Tasas de Fraude
```

Los Feature Flags permiten introducir cambios de forma controlada.

# Categorías de Feature Flags

## Release Flags

Permiten liberar funcionalidades nuevas.

### Ejemplos

```text
new-risk-engine
behavior-profiling-v2
adaptive-authentication-v2
```

---

## Operational Flags

Permiten modificar comportamientos operativos.

### Ejemplos

```text
enable-redis-cache
enable-fallback-authentication
enable-risk-replay
```

---

## Experiment Flags

Permiten realizar pruebas controladas.

### Ejemplos

```text
new-risk-model-a
new-risk-model-b
```

---

## Kill Switches

Permiten deshabilitar funcionalidades inmediatamente.

### Ejemplos

```text
disable-otp-provider
disable-risk-engine-v2
disable-velocity-detection
```

# Integración con Canary Release

Los Feature Flags complementan la estrategia Canary.

---

## Canary

Controla:

```text
Versión desplegada
```

---

## Feature Flags

Controlan:

```text
Funcionalidad habilitada
```

---

## Ejemplo

```text
Canary 10%
        ↓
Feature Flag OFF
```

La funcionalidad aún no está activa.

---

Posteriormente:

```text
Canary 10%
        ↓
Feature Flag ON
```

La funcionalidad comienza a utilizarse.

# Riesgos

## Exceso de Flags

Demasiados flags incrementan complejidad.

### Mitigación

Retiro periódico.

---

## Flags Huérfanas

Flags nunca eliminadas.

### Mitigación

Revisión trimestral.

---

## Dependencias entre Flags

Flags dependientes generan comportamientos difíciles de predecir.

### Mitigación

Evitar dependencias cruzadas.

# Conclusión

La estrategia de Feature Flags permite desacoplar despliegue y activación, reduciendo significativamente el riesgo asociado a cambios en componentes críticos de fraude y autenticación. La combinación de Unleash, Canary Releases y observabilidad continua proporciona una base sólida para evolucionar la plataforma de forma segura, controlada y alineada con prácticas modernas de ingeniería de software y DevOps.
