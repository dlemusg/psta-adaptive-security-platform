# Estrategia de Despliegue

## Propósito

Este documento describe la estrategia de despliegue adoptada para la Plataforma de Seguridad Transaccional Adaptativa (PSTA).

La estrategia busca garantizar que las nuevas versiones puedan ser liberadas de forma:

- Segura.
- Gradual.
- Observable.
- Reversible.
- Compatible con alta disponibilidad.

La solución aprovecha la plataforma corporativa basada en:

```text
Azure DevOps
+
AWS Cloud
+
Kubernetes (EKS)
```

permitiendo automatizar completamente el ciclo de vida de entrega de software.

---

# Objetivos

La estrategia de despliegue debe permitir:

- Minimizar riesgos en producción.
- Reducir downtime.
- Detectar problemas tempranamente.
- Automatizar rollback.
- Liberar funcionalidades progresivamente.
- Mantener continuidad operativa.
- Desplegar microservicios de manera independiente.

---

# Principios

## Zero Downtime

Los despliegues no deben generar indisponibilidad percibida por el cliente.

---

## Progressive Delivery

Las nuevas versiones deben exponerse progresivamente.

---

## Observability First

Toda liberación debe estar acompañada de monitoreo y trazabilidad.

---

## Rollback Fast

Toda liberación debe poder revertirse rápidamente.

---

## Infrastructure as Code

Toda la infraestructura debe ser reproducible mediante código.

---

# Ambientes

## Desarrollo (DEV)

### Objetivo

Validación temprana de funcionalidades durante el desarrollo.

### Características

```text
Despliegue automático
Alta frecuencia de cambios
Validaciones técnicas básicas
```

### Validaciones

- Unit Tests.
- Smoke Tests.
- Pruebas de integración tempranas.

---

## Calidad (QA)

### Objetivo

Certificación funcional y técnica antes de producción.

### Características

```text
Ambiente estable
Datos controlados
Certificación previa a producción
```

### Validaciones

- Integration Tests.
- Contract Tests.
- End-to-End Tests.
- Security Tests.
- Performance Tests básicos.
- Validación funcional por QA.

### Resultado Esperado

```text
Aprobación para Producción
```

---

## Producción (PROD)

### Objetivo

Disponibilizar funcionalidades a usuarios finales con el menor riesgo posible.

### Características

```text
Canary Release
Feature Flags
Observabilidad completa
Rollback automático
```

### Validaciones

- Health Checks.
- Métricas técnicas.
- Métricas de negocio.
- Alertas operativas.
# Estrategia de Rolling Update

Kubernetes realiza actualizaciones graduales de pods.

## Funcionamiento

```text
3 Pods v1

↓

1 Pod v2
2 Pods v1

↓

2 Pods v2
1 Pod v1

↓

3 Pods v2
```

---

## Beneficios

- Sin downtime.
- Menor riesgo.
- Recuperación rápida.

---

# Estrategia Canary Release

Para minimizar riesgos, los despliegues productivos utilizan liberación progresiva.

---

## Fase 1

```text
5% del tráfico
95% versión estable
```

Duración:

```text
15 minutos
```

---

## Fase 2

```text
25% del tráfico
75% versión estable
```

Duración:

```text
30 minutos
```

---

## Fase 3

```text
50% del tráfico
50% versión estable
```

Duración:

```text
60 minutos
```

---

## Fase 4

```text
100% del tráfico
```

Liberación completa.


# Rollback Automático

Si durante el despliegue se detecta:

```text
Incremento anormal de errores
```

o

```text
Degradación significativa del servicio
```

el pipeline ejecuta automáticamente:

```text
Rollback
↓
Versión Estable
```

# Decisión

Se adopta una estrategia basada en:

```text
Azure DevOps
+
Azure Pipelines
+
AWS EKS
+
Canary Releases
+
Feature Flags
```

porque proporciona el mejor equilibrio entre:

- Seguridad.
- Disponibilidad.
- Velocidad de entrega.
- Observabilidad.
- Continuidad operativa.

---


La estrategia de despliegue propuesta aprovecha Azure DevOps como plataforma de automatización y AWS como plataforma de ejecución. La combinación de Azure Pipelines, AWS EKS, Canary Releases y Feature Flags permite realizar despliegues seguros, progresivos y observables, reduciendo significativamente el riesgo asociado a cambios en una plataforma crítica para la prevención de fraude financiero.
