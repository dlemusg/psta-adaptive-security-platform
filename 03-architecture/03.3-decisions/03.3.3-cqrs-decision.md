# Decisión Arquitectónica: Uso de CQRS

# Contexto

La Plataforma de Seguridad Transaccional Adaptativa (PSTA) debe soportar dos tipos de cargas claramente diferenciadas:

## Escrituras (Commands)

Operaciones que modifican el estado del sistema:

- Recepción de transacciones.
- Evaluación de riesgo.
- Autenticaciones.
- Bloqueos de cuentas.
- Actualización de perfiles.

---

## Consultas (Queries)

Operaciones orientadas a lectura:

- Historial de riesgo de un cliente.
- Investigación de fraude.
- Búsqueda de eventos de auditoría.
- Consultas operativas.
- Dashboards.
- Trazabilidad de decisiones.

---

# Problema

Los patrones de acceso de lectura y escritura son significativamente diferentes.

Por ejemplo:

## Escritura

Una transacción debe ser procesada con:

- Baja latencia.
- Alta concurrencia.
- Garantías transaccionales.

```text
TransactionReceived
```

---

## Consulta

Un analista puede solicitar:

```text
Historial de fraude
últimos 12 meses
```

lo que implica:

- Consultas complejas.
- Agregaciones.
- Búsquedas históricas.

---

Intentar resolver ambos escenarios sobre el mismo modelo genera:

- Acoplamiento.
- Modelos complejos.
- Problemas de rendimiento.
- Escalabilidad limitada.

---

# Decisión

Se adopta CQRS (Command Query Responsibility Segregation) de manera selectiva en aquellos dominios donde los patrones de lectura y escritura presentan necesidades claramente diferenciadas.

La solución no implementa CQRS de forma obligatoria para todos los servicios.

Se aplica únicamente donde aporta valor.

---

# Motivación

El principal objetivo es desacoplar:

```text
Modelo Transaccional
```

de

```text
Modelo de Consulta
```

permitiendo optimizar cada uno según sus necesidades.

---

# Aplicación en la Solución

## Risk Assessment

### Command Side

Procesa:

```text
TransactionReceived
AuthenticationCompleted
DeviceTrustChanged
```

Genera:

```text
RiskScoreCalculated
FraudPatternDetected
```

---

### Query Side

Permite consultar:

```text
Historial de riesgo
Tendencias
Patrones sospechosos
Clientes de alto riesgo
```

---

## Audit Service

### Command Side

Consume eventos:

```text
TransactionApproved
TransactionRejected
AuthenticationCompleted
AccountBlocked
```

---

### Query Side

Permite:

```text
Investigaciones
Auditorías
Búsquedas forenses
Consultas regulatorias
```

    ▼

# Modelo Adoptado

## Write Model

Optimizado para:

- Consistencia.
- Reglas de negocio.
- Transacciones.

Tecnología principal:

```text
PostgreSQL
```

---

## Read Model

Optimizado para:

- Búsquedas.
- Agregaciones.
- Consultas complejas.

Tecnología principal:

```text
OpenSearch
```

---

# Flujo de Sincronización

La sincronización se realiza mediante eventos.

```text
Write Model
      │
      ▼

Domain Event

      │
      ▼

Kafka

      │
      ▼

Read Model Builder

      │
      ▼

Read Model
```

---

# Beneficios

## Escalabilidad

Lecturas y escrituras escalan independientemente.

---

## Rendimiento

Cada modelo se optimiza para su propósito.

---

## Simplicidad del Dominio

El modelo transaccional permanece enfocado en reglas de negocio.

---

## Flexibilidad

Nuevos modelos de consulta pueden incorporarse sin afectar el dominio.

Por ejemplo:

```text
Fraud Dashboard
ML Feature Store
Customer Risk View
Executive Dashboard
```

---

# Comparación con Modelo CRUD Tradicional

## CRUD

```text
Application
      │
      ▼

Single Database

      │
      ▼

Reads + Writes
```

### Problemas

- Contención.
- Consultas complejas sobre tablas transaccionales.
- Escalabilidad limitada.

---

## CQRS

```text
Commands
      │
      ▼

Write Model

      │
      ▼

Kafka

      │
      ▼

Read Models

      │
      ▼

Queries
```

### Beneficios

- Desacoplamiento.
- Rendimiento.
- Escalabilidad.

---

# Alternativas Evaluadas

## CRUD Tradicional

### Ventajas

- Simplicidad.

### Desventajas

- Escalabilidad limitada.
- Consultas complejas impactan la operación.

### Decisión

Descartada para dominios analíticos.

---

## Data Warehouse Exclusivo

### Ventajas

- Potencia analítica.

### Desventajas

- Latencia elevada.
- No adecuado para consultas operativas.

### Decisión

Complementario, no sustituto.

---

# Riesgos Introducidos

## Mayor Complejidad

Aparecen:

- Read Models.
- Proyecciones.
- Sincronización.

---

## Consistencia Eventual

El modelo de lectura puede presentar pequeños retrasos.

---

## Operación

Se requiere monitorear:

- Proyecciones.
- Consumidores.
- Sincronización.

---

# Mitigaciones

## Event-Driven Architecture

Kafka actúa como mecanismo de sincronización.

---

## Idempotencia

Las proyecciones deben soportar reprocesamiento.

---

## Observabilidad

Se monitorean:

- Lag de consumidores.
- Estado de proyecciones.
- Errores de sincronización.

---

# Relación con los Objetivos del Reto

| Objetivo | Contribución de CQRS |
|-----------|----------------------|
| Detección de fraude | Consultas optimizadas de historial |
| Investigación | Modelos especializados de búsqueda |
| Auditoría | Consultas eficientes sobre eventos |
| Escalabilidad | Separación de lectura y escritura |
| Evolución futura | Incorporación de nuevas vistas sin afectar el dominio |

---

# ¿Por qué CQRS Parcial y no Total?

La solución adopta CQRS únicamente en dominios donde existe una diferencia significativa entre lectura y escritura.

No todos los microservicios justifican el costo adicional.


| Servicio | CQRS |
|-----------|------|
| Risk Assessment | Sí |
| Audit | Sí |
| Transaction Decision | No |
| Authentication | No |
| Device Trust | No |

Esto permite mantener un balance adecuado entre complejidad y beneficio.

---



Se adopta CQRS de forma selectiva para separar las responsabilidades de lectura y escritura en dominios donde los patrones de acceso son significativamente diferentes.

Esta decisión mejora la escalabilidad, facilita las investigaciones de fraude y permite construir modelos de consulta especializados sin comprometer el rendimiento ni la simplicidad del dominio transaccional.
