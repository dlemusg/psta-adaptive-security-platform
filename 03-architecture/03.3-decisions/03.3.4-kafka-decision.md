# Decisión Arquitectónica: Uso de Apache Kafka


# Contexto

La Plataforma de Seguridad Transaccional Adaptativa (PSTA) debe soportar procesamiento de eventos en tiempo real asociados a:

- Transacciones financieras.
- Evaluaciones de riesgo.
- Autenticaciones adaptativas.
- Detección de fraude.
- Bloqueos de cuentas.
- Auditoría y cumplimiento.

Cada evento generado por el sistema puede ser consumido por múltiples dominios de negocio simultáneamente.

Por ejemplo:

```text
TransactionReceived
```

puede ser utilizado por:

- Risk Assessment Service.
- Audit Service.
- Customer Profile Service.
- Analytics.
- Futuros modelos de Machine Learning.

---

# Problema

La solución requiere una plataforma de integración que permita:

- Desacoplamiento entre servicios.
- Escalabilidad horizontal.
- Reprocesamiento de eventos.
- Trazabilidad completa.
- Persistencia temporal de eventos.
- Múltiples consumidores independientes.

Las alternativas tradicionales basadas exclusivamente en colas presentan limitaciones importantes para este escenario.

---

# Decisión

Se adopta Apache Kafka como backbone principal de eventos para la plataforma.

Kafka será utilizado para:

- Publicación de eventos de dominio.
- Integración entre microservicios.
- Construcción de modelos CQRS.
- Auditoría.
- Procesamiento en tiempo real.

---

# Motivación

La naturaleza del problema corresponde a un escenario de Event Streaming y no únicamente a distribución de trabajo (Work Queue).

El objetivo no es simplemente entregar un mensaje a un consumidor.

El objetivo es permitir que múltiples dominios reaccionen de forma independiente a los mismos eventos de negocio.

---

# Ejemplo de Negocio

Cuando una transacción es recibida:

```text
TransactionReceived
```

los siguientes servicios pueden reaccionar simultáneamente:

```text
Risk Assessment
Audit
Customer Profile
Analytics
Future Consumers
```

Cada consumidor procesa el evento sin afectar a los demás.

---

# Arquitectura Conceptual

```text
                  Kafka

                    │

      ┌─────────────┼─────────────┐
      ▼             ▼             ▼

Risk Assessment   Audit     Customer Profile

      ▼             ▼             ▼

Consumer Group Consumer Group Consumer Group
```

Cada consumidor mantiene su propio offset.

---

# ¿Por Qué Kafka y No una Cola Tradicional?

## Modelo de Cola

Una cola tradicional sigue normalmente el patrón:

```text
Producer
    │
    ▼

 Queue

    │
    ▼

Consumer
```

La semántica principal es:

```text
1 mensaje
↓
1 consumidor
```

Una vez procesado:

```text
Mensaje eliminado
```

---

# Limitaciones de las Colas Tradicionales

## Multiplicidad de Consumidores

Un evento debe ser consumido por múltiples servicios.

Con una cola tradicional suelen requerirse:

- Fan-Out.
- Exchanges.
- Duplicación de mensajes.
- Configuración adicional.

La complejidad aumenta a medida que aparecen nuevos consumidores.

---

## Reprocesamiento

Un nuevo modelo de fraude puede requerir:

```text
Recalcular
últimos 30 días
```

Con una cola tradicional:

```text
Mensaje eliminado
```

No existe historial disponible.

---

## Auditoría

La solución debe soportar:

```text
¿Por qué se aprobó?
¿Por qué se rechazó?
¿Qué score tenía?
¿Qué autenticación se utilizó?
```

Las colas tradicionales no fueron diseñadas para mantener un historial completo de eventos.

---

# Ventajas de Kafka

## Event Streaming

Kafka fue diseñado para procesamiento continuo de eventos.

Permite:

```text
Publicar
Consumir
Reprocesar
Analizar
```

sobre el mismo flujo de eventos.

---

## Retención de Eventos

Los eventos permanecen almacenados durante un período configurable.

Por ejemplo:

```text
7 días
30 días
90 días
```

o incluso indefinidamente.

---

## Reprocesamiento

Un consumidor puede volver a procesar eventos históricos simplemente reiniciando su offset.

```text
Offset = 0
```

---

## Múltiples Consumidores

Cada consumidor mantiene independencia total.

```text
Risk Assessment
Offset 1500

Audit
Offset 1495

Analytics
Offset 1200
```

---

## Escalabilidad Horizontal

Kafka distribuye eventos mediante particiones.

```text
Topic
 ├── Partition 1
 ├── Partition 2
 ├── Partition 3
 └── Partition N
```

Esto permite:

- Paralelismo.
- Escalabilidad.
- Balanceo de carga.

---

## Persistencia

Kafka actúa como registro distribuido de eventos.

Permite reconstruir el historial completo del sistema.

---

# Relación con CQRS

Kafka es el mecanismo utilizado para sincronizar los modelos de lectura.

```text
Write Model
      │
      ▼

Domain Event

      │
      ▼

Kafka

      │
      ▼

Read Model Builder

      │
      ▼

OpenSearch
```

---

# Relación con la Detección de Fraude

El motor de fraude requiere analizar eventos en tiempo real.

Ejemplos:

```text
TransactionReceived
AuthenticationCompleted
BeneficiaryAdded
DeviceChanged
```

Kafka permite construir un flujo continuo de señales para alimentar:

```text
Velocity Engine
Fraud Rule Engine
Risk Score Calculator
```

---

# Relación con Auditoría

Todos los eventos relevantes permanecen disponibles para:

- Investigación.
- Cumplimiento regulatorio.
- Reconstrucción histórica.
- Análisis forense.

---

# Alternativas Evaluadas

## RabbitMQ

### Ventajas

- Excelente para Work Queues.
- Simplicidad.
- Baja latencia.

### Desventajas

- Reprocesamiento limitado.
- Retención limitada.
- Menor capacidad para Event Streaming.

### Decisión

Descartado como backbone principal.

---

## IBM MQ

### Ventajas

- Alta robustez.
- Amplia adopción bancaria.

### Desventajas

- Orientado principalmente a mensajería tradicional.
- Menor flexibilidad para streaming.

### Decisión

Descartado para este escenario.

---

## REST Síncrono

### Ventajas

- Simplicidad conceptual.

### Desventajas

- Acoplamiento temporal.
- Dependencia de disponibilidad.
- Escalabilidad limitada.

### Decisión

Utilizado únicamente para integraciones específicas.

---

# Riesgos Introducidos

## Complejidad Operacional

Kafka requiere:

- Administración de clúster.
- Gobierno de eventos.
- Monitoreo.

---

## Gestión de Esquemas

Los contratos deben evolucionar de forma controlada.

---

## Idempotencia

Los consumidores deben tolerar reprocesamiento.

---

# Mitigaciones

## Schema Registry

Permite versionar contratos de eventos.

### Formato seleccionado

```text
Apache Avro
```

---

## Outbox Pattern

Garantiza consistencia entre:

```text
Base de datos
Evento publicado
```

---

## Dead Letter Topics

Permiten gestionar errores sin perder eventos.

---

## Consumer Groups

Permiten escalado horizontal seguro.

# Relación con los Objetivos del Reto

| Objetivo | Contribución de Kafka |
|-----------|-----------------------|
| Detección temprana de fraude | Procesamiento continuo de eventos |
| Fraude por velocidad | Streaming de transacciones |
| Resiliencia | Desacoplamiento de servicios |
| Escalabilidad | Particiones y Consumer Groups |
| Auditoría | Persistencia de eventos |
| Evolución futura | Nuevos consumidores sin modificar productores |

---

# Consecuencias

## Positivas

- Desacoplamiento.
- Escalabilidad.
- Reprocesamiento.
- Auditabilidad.
- Flexibilidad.

---

## Negativas

- Mayor complejidad operacional.
- Necesidad de gobierno de eventos.
- Curva de aprendizaje.

---

# Conclusión

Apache Kafka fue seleccionado como backbone de integración debido a que el problema planteado corresponde a un escenario de Event Streaming y no únicamente a distribución de mensajes.

La capacidad de soportar múltiples consumidores independientes, retención de eventos, reprocesamiento histórico, auditoría y escalabilidad horizontal convierte a Kafka en la alternativa más adecuada para una plataforma antifraude bancaria basada en eventos y procesamiento en tiempo real.
