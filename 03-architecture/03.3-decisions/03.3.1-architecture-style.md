# Decisión Arquitectónica: Estilo Arquitectónico

# Contexto

La Plataforma de Seguridad Transaccional Adaptativa (PSTA) debe resolver un conjunto de desafíos característicos de una plataforma antifraude bancaria moderna:

- Procesamiento transaccional en tiempo real.
- Detección temprana de fraude.
- Autenticación adaptativa.
- Integración con múltiples proveedores externos.
- Alta disponibilidad.
- Escalabilidad horizontal.
- Trazabilidad completa.
- Evolución continua de capacidades de negocio.

Adicionalmente, la solución debe soportar crecimiento futuro sin generar dependencias excesivas entre los diferentes dominios funcionales.

---

# Problema

Una arquitectura tradicional monolítica o basada principalmente en integraciones síncronas genera riesgos importantes:

- Alto acoplamiento.
- Escalabilidad limitada.
- Dependencias operativas entre módulos.
- Dificultad para evolucionar capacidades de negocio de manera independiente.
- Mayor impacto ante fallos de sistemas externos.

La plataforma requiere un enfoque arquitectónico que permita responder rápidamente a eventos de negocio, escalar selectivamente y mantener resiliencia operacional.

---

# Decisión

Se adopta una arquitectura basada en la combinación de:

- Domain-Driven Design (DDD).
- Event-Driven Architecture (EDA).
- Microservicios.
- Arquitectura Reactiva.
- Consistencia Eventual.

La solución se estructura alrededor de capacidades de negocio independientes comunicadas principalmente mediante eventos.

---

# Estilos Arquitectónicos Seleccionados

## Domain-Driven Design (DDD)

La solución se organiza alrededor de dominios de negocio claramente definidos.

### Bounded Contexts

- Transaction Decision
- Risk Assessment
- Adaptive Authentication
- Account Protection
- Device Trust
- Customer Profile
- Audit

### Razón

Cada dominio posee:

- Reglas propias.
- Modelos propios.
- Ciclos de evolución diferentes.

DDD permite mantener dichas capacidades desacopladas.

---

## Event-Driven Architecture

La comunicación entre servicios se realiza principalmente mediante eventos de negocio.

### Backbone

```text
Apache Kafka
```

### Eventos representativos

```text
TransactionReceived
RiskScoreCalculated
AuthenticationCompleted
FraudPatternDetected
AccountBlocked
```

### Razón

El negocio es naturalmente orientado a eventos.

Cada transacción, autenticación o alerta genera información que debe ser procesada por múltiples dominios.

---

## Microservicios

Cada capacidad de negocio se implementa como un servicio independiente.

### Servicios principales

```text
Transaction Decision Service
Risk Assessment Service
Adaptive Authentication Service
Account Protection Service
Device Trust Service
Customer Profile Service
Audit Service
```

### Razón

Permitir:

- Escalado independiente.
- Despliegues independientes.
- Equipos autónomos.
- Evolución independiente.

---

## Arquitectura Reactiva

Los servicios se implementan utilizando:

```text
Java 21
Spring WebFlux
Project Reactor
```

### Razón

La plataforma debe soportar:

- Miles de eventos concurrentes.
- Baja latencia.
- Alta utilización de recursos.

La programación reactiva reduce el consumo de hilos y mejora el rendimiento bajo carga.

---

## Consistencia Eventual

La plataforma prioriza:

```text
Disponibilidad
Escalabilidad
Resiliencia
```

sobre consistencia inmediata entre dominios.

### Razón

Los distintos contextos mantienen información distribuida.

Intentar sincronización inmediata incrementaría significativamente el acoplamiento.

# Alternativas Evaluadas

| Alternativa Evaluada | Ventajas | Desventajas | Decisión |
| :--- | :--- | :--- | :--- |
| **Monolito** | - Menor complejidad inicial.<br>- Despliegue sencillo. | - Escalabilidad limitada.<br>- Acoplamiento elevado.<br>- Evolución compleja. | **Descartada.** |
| **SOA Tradicional** | - Separación funcional.<br>- Reutilización. | - Dependencia excesiva de integración síncrona.<br>- Mayor fragilidad operacional. | **Descartada.** |
| **Arquitectura Basada en REST Exclusivamente** | - Simplicidad conceptual. | - Acoplamiento temporal.<br>- Menor resiliencia.<br>- Dificultad para procesar eventos masivos. | **Descartada como mecanismo principal de integración.** |



---

# Consecuencias

## Positivas

| Dimensión / Atributo | Ventajas Positivas | Características |
| :--- | :--- | :--- |
| **Escalabilidad** | Escalado independiente | Cada servicio puede escalar de manera autónoma según su propia demanda y carga de trabajo. |
| **Resiliencia** | Tolerancia a fallos aislada | La caída de un componente no implica necesariamente la caída de toda la plataforma. |
| **Evolución Independiente** | Desacoplamiento de dominios | Los dominios pueden evolucionar y desplegarse sin afectar otros contextos de negocio. |
| **Auditabilidad** | Historial de decisiones | Los eventos permiten reconstruir completamente y de forma inmutable el historial de decisiones. |
| **Flexibilidad** | Extensibilidad de capacidades | La arquitectura facilita la incorporación futura de nuevas capacidades, tales como:<br>- Motores de Machine Learning.<br>- Nuevos factores de autenticación.<br>- Nuevas estrategias de mitigación. |


---

## Negativas

| Desafío / Trade-off | Impacto | Requerimientos / Implicaciones |
| :--- | :--- | :--- |
| **Mayor Complejidad Operacional** | La infraestructura distribuida incrementa el esfuerzo de administración y mantenimiento. | Requiere:<br>- Observabilidad avanzada.<br>- Automatización.<br>- Gobierno de eventos. |
| **Consistencia Eventual** | Existe un desfase temporal inherente en la propagación de datos a través del sistema. | Algunas actualizaciones no son visibles inmediatamente en todos los dominios. |
| **Curva de Aprendizaje** | El equipo de desarrollo e infraestructura se enfrenta a paradigmas avanzados de diseño. | DDD, Kafka y programación reactiva requieren conocimiento especializado. |
---

# Relación con los Objetivos del Reto

| Objetivo | Estilo Arquitectónico que lo soporta |
|-----------|--------------------------------------|
| Detección temprana de fraude | Event-Driven Architecture |
| Fraude por velocidad | Arquitectura Reactiva |
| Escalabilidad | Microservicios |
| Resiliencia | Event-Driven Architecture |
| Evolución independiente | DDD |
| Auditabilidad | Eventos de Dominio |
| Continuidad operativa | Arquitectura Distribuida |

---


Se adopta una arquitectura basada en Domain-Driven Design, Event-Driven Architecture y Microservicios Reactivos debido a que proporciona el mejor equilibrio entre escalabilidad, resiliencia, auditabilidad y capacidad de evolución para una plataforma antifraude bancaria.

Esta decisión permite que la solución responda eficientemente a eventos de negocio en tiempo real, reduzca el acoplamiento entre capacidades y mantenga la flexibilidad necesaria para incorporar nuevas funcionalidades en el futuro.
